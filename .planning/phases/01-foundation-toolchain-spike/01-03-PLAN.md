**src/auth/session.mpl** — PostgreSQL-backed session store (Claude's discretion: PG sessions over Valkey per research Open Question #1 — Mesh may lack a Valkey client):

`fn create_session(pool, user_id :: String) -> String!String`:
1. Generate session ID with `Crypto.uuid4()`
2. Set expires_at to NOW() + 24 hours (use SQL: `NOW() + INTERVAL '24 hours'`)
3. INSERT INTO sessions (id, user_id, expires_at) VALUES ($1, $2, NOW() + INTERVAL '24 hours')
4. Return the session ID (this becomes the cookie value)

`fn validate_session(pool, session_id :: String) -> Option<Map<String, String>>`:
1. Query: `SELECT s.id, s.user_id, u.email FROM sessions s JOIN users u ON s.user_id = u.id WHERE s.id = $1 AND s.expires_at > NOW()`
2. Return Some(first_row) if found, None if not found or expired
3. This returns user info alongside session validation in a single query

`fn destroy_session(pool, session_id :: String) -> Int!String`:
1. DELETE FROM sessions WHERE id = $1
2. Return rows affected

`fn destroy_user_sessions(pool, user_id :: String) -> Int!String`:
1. DELETE FROM sessions WHERE user_id = $1
2. Used for password reset (invalidate all sessions) and logout-everywhere

`fn cleanup_expired(pool) -> Int!String`:
1. DELETE FROM sessions WHERE expires_at < NOW()
2. Called periodically (can be triggered on login or by a background actor later)

**src/auth/middleware.mpl** — HTTP middleware:

`fn auth_middleware(request :: Request, next) -> Response`:
1. Extract cookie header: `Request.header(request, "cookie")`
2. Parse `mesher_session=<value>` from cookie string (manual parse — split on `;`, trim, find key)
3. If no cookie or no mesher_session key: return `HTTP.response(401, json { error: "unauthorized" })`
4. Validate session: call `validate_session(pool, session_id)` — pool must be accessible (import or pass via closure)
5. If session invalid/expired: return 401
6. If valid: call `next(request)` to continue chain
7. NOTE: Mesh middleware currently cannot inject arbitrary context into the request object. The user_id will need to be passed through a mechanism — options:
   a. Store user context in an actor mailbox/process dictionary if available
   b. Re-query user from session in handlers that need it
   c. Use a request header injection pattern if the runtime supports it
   For now, handlers that need user context should call validate_session themselves with the session cookie. Document this as a known pattern until Mesh adds request context injection.

`fn tier_gate(request :: Request, next) -> Response`:
1. Check `Config.tier()` — if "oss", block SaaS-only routes
2. If path starts with "/api/v1/ai/" and tier != "saas": return `HTTP.response(403, json { error: "SaaS only" })`
3. Otherwise: `next(request)`

Both middleware must handle the cookie string being absent (None case from Request.header).

**src/auth/handlers.mpl** — Auth HTTP handlers:

`fn register_handler(request :: Request) -> Response` (POST /api/auth/register):
1. Parse JSON body: extract `email` and `password` fields from `Request.body(request)` using `Json.parse`
2. Validate: email is non-empty, password is non-empty (minimum validation)
3. Check uniqueness: `SELECT id FROM users WHERE email = $1` — if exists, return `HTTP.response(409, json { error: "email already registered" })`
4. Generate salt: `Crypto.uuid4()`
5. Hash password: `Crypto.hmac_sha256(Config.session_secret() <> salt, password)` — per research recommendation (HMAC-SHA256 with server secret + per-user salt)
6. Generate user ID: `Crypto.uuid4()`
7. INSERT INTO users (id, email, password_hash, password_salt) VALUES ($1, $2, $3, $4)
8. Create session: call `Session.create_session(pool, user_id)`
9. Return `HTTP.response(201, json { id: user_id, email: email })` with `Set-Cookie: mesher_session={session_id}; HttpOnly; SameSite=Strict; Path=/; Max-Age=86400`

`fn login_handler(request :: Request) -> Response` (POST /api/auth/login):
1. Parse JSON body: extract `email` and `password`
2. Query user: `SELECT id, password_hash, password_salt FROM users WHERE email = $1`
3. If not found: return `HTTP.response(401, json { error: "invalid credentials" })`
4. Verify password: compute `Crypto.hmac_sha256(Config.session_secret() <> salt, password)` and compare with stored hash using `Crypto.secure_compare(computed, stored)` — constant-time comparison
5. If mismatch: return 401
6. Clean up expired sessions: call `Session.cleanup_expired(pool)` (opportunistic cleanup)
7. Create session: call `Session.create_session(pool, user_id)`
8. Return `HTTP.response(200, json { id: user_id, email: email })` with Set-Cookie header

`fn logout_handler(request :: Request) -> Response` (POST /api/auth/logout):
1. Extract session cookie from request (same parsing as middleware)
2. If no cookie: return 401
3. Destroy session: `Session.destroy_session(pool, session_id)`
4. Return `HTTP.response(200, json { status: "logged out" })` with `Set-Cookie: mesher_session=; HttpOnly; SameSite=Strict; Path=/; Max-Age=0` (clear cookie)

`fn me_handler(request :: Request) -> Response` (GET /api/auth/me):
1. Extract and validate session (re-query since middleware can't inject context)
2. Return `HTTP.response(200, json { id: user_id, email: email })`

**Set-Cookie header construction:** Build the cookie string manually:
```mesh
let cookie = "mesher_session=" <> session_id <> "; HttpOnly; SameSite=Strict; Path=/; Max-Age=86400"
```
Use a helper function to add the Set-Cookie header to the response. If Mesh HTTP.response doesn't support custom headers, check if there's a Response builder pattern. If not available, document this as a potential Mesh limitation that needs resolution.

**Update src/main.mpl** — Wire auth routes:
1. Import auth handlers and middleware
2. Add routes BEFORE auth_middleware (public routes):
   - `HTTP.on_post(r, "/api/auth/register", register_handler)`
   - `HTTP.on_post(r, "/api/auth/login", login_handler)`
3. Add auth middleware: `HTTP.use(r, auth_middleware)`
4. Add routes AFTER auth_middleware (protected routes):
   - `HTTP.on_post(r, "/api/auth/logout", logout_handler)`
   - `HTTP.on_get(r, "/api/auth/me", me_handler)`
5. Add tier_gate middleware before all routes

IMPORTANT: Mesh's `HTTP.use` middleware applies in registration order. Routes registered before `HTTP.use(r, auth_middleware)` are NOT protected. Routes after ARE protected. This is the mechanism for public vs authenticated endpoints.
