## Summary

The Mesh language includes a built-in ORM with three primary subsystems: (1) a **Migration DSL** (`Migration.create_table` / `Migration.drop_table`) with a CLI runner (`meshc migrate up/down/status/generate`), (2) a **Repo module** providing CRUD operations (`Repo.insert`, `Repo.get`, `Repo.get_by`, `Repo.delete`, `Repo.update_where`, `Repo.delete_where`, `Repo.query_raw`, `Repo.execute_raw`), and (3) a **Query builder** (`Query.from`, `Query.where`, `Query.where_raw`, `Query.select_raw`, `Query.join_as`, `Query.order_by`, `Query.limit`, `Query.group_by_raw`, `Query.order_by_raw`) for composable queries.

The current mesher project uses raw SQL exclusively: `Pool.query`/`Pool.execute` for public schema operations, and `Pg.query`/`Pg.execute` inside `with_org_schema` for tenant-scoped operations. The reference project at `/Users/sn0w/Documents/dev/mesh/mesher` demonstrates comprehensive ORM usage with clearly documented "ORM boundaries" where raw SQL is intentionally retained for operations the ORM cannot express (complex UPSERT, server-side JSONB extraction, multi-table subqueries, PG function calls in SET clauses).

**Primary recommendation:** Follow the reference project's architecture exactly: create `types/` directory for model structs with `deriving(Schema, Json, Row)`, create a centralized `storage/queries.mpl` module for all data access, rewrite migrations using `Migration.create_table`, and replace `Pool.query`/`Pool.execute` with `Repo.*` and `Query.*` calls. The current project's schema-per-org tenant isolation is a significant architectural difference from the reference project (which uses a flat single-schema model), requiring special handling.

### Pattern 1: Model Definition with `deriving(Schema, Json, Row)`
**What:** Define structs with `table` annotation and relationship declarations. The `deriving(Schema, Json, Row)` enables ORM integration: `Schema` provides `__table__()` for Query.from, `Row` enables automatic row mapping, `Json` enables serialization.
**When to use:** Every database entity needs a corresponding struct.
**Example:**
```mesh
# Source: /Users/sn0w/Documents/dev/mesh/mesher/types/user.mpl
pub struct User do
  table "users"
  id :: String
  email :: String
  display_name :: String
  created_at :: String
  has_many :org_memberships, OrgMembership
  has_many :sessions, Session
end deriving(Schema, Json, Row)
```

**Critical observations:**
- ALL fields are `String` type (even timestamps, UUIDs, booleans) -- the Row deriving maps through `Map<String, String>` text protocol
- `Bool` type IS supported (see `AlertRule.enabled :: Bool` in reference)
- `Int` type IS supported (see `Issue.event_count :: Int` in reference)
- `Option<String>` IS supported for nullable columns (see `ApiKey.revoked_at :: Option<String>`)
- `primary_key :field_name` overrides default `id` PK (see `Session.primary_key :token`)
- Sensitive fields can be omitted from struct (see User -- no `password_hash` field)
- `has_many` / `belongs_to` declare relationships but are not used for eager loading -- they serve as documentation and potential future ORM features

### Pattern 2: Centralized Query Module
**What:** All data access functions live in a single `storage/queries.mpl` module, taking `pool :: PoolHandle` as first argument. Services and handlers import specific functions.
**When to use:** Always. Handlers should never contain SQL strings.
**Example:**
```mesh
# Source: /Users/sn0w/Documents/dev/mesh/mesher/storage/queries.mpl
pub fn get_org(pool :: PoolHandle, id :: String) -> Organization!String do
  let row = Repo.get(pool, Organization.__table__(), id)?
  Ok(Organization { id: Map.get(row, "id"), name: Map.get(row, "name"), slug: Map.get(row, "slug"), created_at: Map.get(row, "created_at") })
end
```

### Pattern 3: ORM CRUD Operations
**What:** Use `Repo.insert`, `Repo.get`, `Repo.get_by`, `Repo.delete` for single-record operations with `Map<String, String>` field maps.
**When to use:** Simple insert/select/delete by primary key or unique field.
**Example:**
```mesh
# Insert -- returns Map<String, String> with generated values (id, created_at, etc.)
let fields = %{"email" => email, "password_hash" => password_hash, "display_name" => display_name}
let row = Repo.insert(pool, User.__table__(), fields)?

# Get by PK
let row = Repo.get(pool, User.__table__(), id)?

# Get by unique field
let row = Repo.get_by(pool, Project.__table__(), "slug", slug)?

# Delete by PK
let _ = Repo.delete(pool, OrgMembership.__table__(), membership_id)?
```

### Pattern 4: Query Builder for Complex Reads
**What:** Compose queries using `Query.from` pipeline with `where`, `where_raw`, `select_raw`, `order_by`, `limit`, `join_as`.
**When to use:** Filtered lists, JOINs, aggregations, conditional WHERE clauses.
**Example:**
```mesh
# Source: /Users/sn0w/Documents/dev/mesh/mesher/storage/queries.mpl
pub fn list_projects_by_org(pool :: PoolHandle, org_id :: String) -> List<Project>!String do
  let q = Query.from(Project.__table__())
    |> Query.where(:org_id, org_id)
    |> Query.order_by(:name, :asc)
  let rows = Repo.all(pool, q)?
  Ok(List.map(rows, fn(row) do
    Project { id: Map.get(row, "id"), ... }
  end))
end
```

### Pattern 5: Update and Delete with Query Conditions
**What:** `Repo.update_where` and `Repo.delete_where` combine Query builder conditions with mutations.
**When to use:** UPDATE/DELETE with WHERE conditions beyond primary key.
**Example:**
```mesh
# Update
let q = Query.from(Issue.__table__())
  |> Query.where_raw("id = ?::uuid", [issue_id])
let _ = Repo.update_where(pool, Issue.__table__(), %{"status" => "resolved"}, q)?

# Delete
let q = Query.from(Session.__table__())
  |> Query.where(:token, token)
Repo.delete_where(pool, Session.__table__(), q)
```

### Pattern 6: Migration DSL
**What:** `Migration.create_table(pool, table_name, columns)` with column specs as `"name:TYPE:CONSTRAINTS"` strings.
**When to use:** All DDL in migration files.
**Example:**
```mesh
# Source: /Users/sn0w/Documents/dev/mesh/mesher/migrations/20260216120000_create_initial_schema.mpl
pub fn up(pool :: PoolHandle) -> Int!String do
  Migration.create_table(pool, "users", [
    "id:UUID:PRIMARY KEY DEFAULT gen_random_uuid()",
    "email:TEXT:NOT NULL UNIQUE",
    "display_name:TEXT:NOT NULL",
    "created_at:TIMESTAMPTZ:NOT NULL DEFAULT now()"
  ])?
  Ok(0)
end

pub fn down(pool :: PoolHandle) -> Int!String do
  Migration.drop_table(pool, "users")?
  Ok(0)
end
```

**Key migration observations:**
- Functions MUST be `pub fn up(pool :: PoolHandle) -> Int!String` and `pub fn down(pool :: PoolHandle) -> Int!String`
- Column format is `"name:TYPE:CONSTRAINTS"` with colons separating parts
- Complex table structures (e.g., composite UNIQUE constraints, FK with ON DELETE CASCADE inline) may still need `Pool.execute` for raw DDL
- Migration files are named `YYYYMMDDHHMMSS_description.mpl` and generated via `meshc migrate generate "description"`
- `meshc migrate up` applies all pending migrations; `meshc migrate down` rolls back the last one
- Migration runner tracks state automatically (no manual tracking needed)

### Pattern 7: Two-Step Pattern for Server-Side Functions
**What:** When ORM cannot express server-side PG functions (like `crypt()`, `now()`), use a two-step approach: `Repo.query_raw` to compute the value, then `Repo.insert`/`Repo.update_where` with the literal result.
**When to use:** bcrypt hashing, server-side timestamps, computed values.
**Example:**
```mesh
# Source: /Users/sn0w/Documents/dev/mesh/mesher/storage/queries.mpl
pub fn create_user(pool :: PoolHandle, email :: String, password :: String, display_name :: String) -> String!String do
  # Step 1: Hash password via pgcrypto
  let hash_rows = Repo.query_raw(pool, "SELECT crypt($1, gen_salt('bf', 12)) AS hash", [password])?
  let password_hash = Map.get(List.head(hash_rows), "hash")
  # Step 2: Insert user with ORM
  let fields = %{"email" => email, "password_hash" => password_hash, "display_name" => display_name}
  let row = Repo.insert(pool, User.__table__(), fields)?
  Ok(Map.get(row, "id"))
end
```

### Anti-Patterns to Avoid
- **SQL in handler files:** Never put `Pool.query`/`Pool.execute`/`Pg.query`/`Pg.execute` in handler files. All data access goes through `storage/queries.mpl`.
- **Deeply nested case blocks for error handling:** Use the `?` operator with `-> Type!String` return annotations to propagate errors. The current codebase has 7+ levels of nesting in `handle_create_org` that should flatten to 3-4 levels.
- **Using `Pool.query`/`Pool.execute` for data queries:** Replace with `Repo.all`, `Repo.get`, `Repo.insert`, etc. Only use `Pool.execute` for DDL that is not in migrations.
- **Manual connection checkout for simple queries:** The Repo module handles connection management internally. Only use `Pool.checkout`/`Pool.checkin`/`Pg.begin`/`Pg.commit`/`Pg.rollback` when you genuinely need manual transaction control (like `with_org_schema`).
- **Duplicating the same SQL query in multiple handlers:** Centralize in `storage/queries.mpl` and import.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Table creation in migrations | `Pool.execute(pool, "CREATE TABLE ...")` | `Migration.create_table(pool, name, columns)` | DSL handles IF NOT EXISTS, consistent format, pairs with `Migration.drop_table` |
| Migration tracking | Manual timestamp-based ordering | `meshc migrate up/down/status` | Built-in migration runner tracks applied state automatically |
| Single-record CRUD | `Pool.query(pool, "SELECT * FROM x WHERE id = $1", [id])` | `Repo.get(pool, Table.__table__(), id)` | Cleaner API, consistent error handling |
| Filtered queries | String-concatenated SQL | `Query.from |> Query.where |> Query.where_raw` | Composable, parameterized, less error-prone |
| INSERT RETURNING | `Pool.query(pool, "INSERT ... RETURNING id", [...])` | `Repo.insert(pool, Table.__table__(), fields)` | Returns full row map automatically |
| Record deletion | `Pool.execute(pool, "DELETE FROM x WHERE id = $1", [id])` | `Repo.delete(pool, Table.__table__(), id)` | Consistent API |

**Key insight:** The ORM covers ~80% of data operations cleanly. The remaining ~20% (server-side function calls, complex UPSERT, JSONB extraction, multi-table subqueries) uses `Repo.query_raw`/`Repo.execute_raw` with clear documentation of WHY raw SQL is needed (see reference project's "ORM boundary" comments).

## Common Pitfalls

### Pitfall 1: UUID Type Casting with `?::uuid`
**What goes wrong:** The ORM's `Query.where(:field, value)` passes string values. When the database column is UUID type, PostgreSQL may fail to match TEXT to UUID.
**Why it happens:** The reference project uses `UUID` column types (with `gen_random_uuid()` defaults) while the current project uses `TEXT` column types for IDs.
**How to avoid:** Two options: (a) Keep TEXT PKs and avoid casting issues entirely, or (b) Switch to UUID PKs and always use `Query.where_raw("id = ?::uuid", [id])` instead of `Query.where(:id, id)` for UUID columns. The reference project consistently uses `?::uuid` casting in `where_raw` for UUID columns.
**Recommendation:** Switch to UUID PKs (matching reference project) since this is a fresh migration start. Use `Query.where_raw` with `?::uuid` casts for all UUID column comparisons.
**Warning signs:** "operator does not exist: uuid = text" errors at runtime.

### Pitfall 2: NULL Values in Map<String, String>
**What goes wrong:** `Repo.update_where` takes `Map<String, String>` for SET values, which cannot represent SQL NULL.
**Why it happens:** The Map type only holds String values -- no way to express "set this column to NULL".
**How to avoid:** Use `Repo.execute_raw` for updates that need to SET columns to NULL. The reference project documents this explicitly (see `assign_issue` -- "unassign branch retains Repo.execute_raw since ORM Map<String,String> cannot represent NULL").
**Warning signs:** Wanting to "unset" a nullable column value.

### Pitfall 3: Server-Side Functions in SET Clauses
**What goes wrong:** `Repo.update_where` takes literal string values in its Map, so `now()`, `crypt()`, and computed expressions cannot be used.
**Why it happens:** The Map<String,String> interface only passes literal values, not SQL expressions.
**How to avoid:** Use the two-step pattern: `Repo.query_raw("SELECT now()::text AS ts", [])` to get the computed value, then `Repo.update_where` with the literal result. Or use `Repo.execute_raw` for the entire UPDATE. The reference project documents these as "ORM boundaries".
**Warning signs:** Needing `SET column = now()` or `SET column = crypt(...)` in updates.

### Pitfall 4: Tenant Schema Isolation with ORM
**What goes wrong:** The current project uses `with_org_schema` (SET LOCAL search_path) within manually managed transactions. The ORM's Repo functions operate on the connection pool directly and don't support `SET LOCAL search_path` within a transaction context.
**Why it happens:** The Repo module manages its own connections from the pool. There is no `Repo.with_connection` or `Repo.with_transaction` API that allows injecting search_path changes.
**How to avoid:** This is the biggest architectural challenge. Options:
  1. **Keep `with_org_schema` but use `Repo.query_raw`/`Repo.execute_raw` inside it** -- retains tenant isolation pattern but uses Repo escape hatches instead of raw `Pg.query`/`Pg.execute`
  2. **Restructure to single-schema with FK-based org scoping** -- matches reference project's flat model (all tables in public schema, `org_id` FK on projects/api_keys). This eliminates `with_org_schema` entirely but changes the data model
  3. **Use manual Pool.checkout + Pg.begin + SET LOCAL + Pg.execute + Pg.commit pattern** -- functionally identical to current approach, just cleaner with ? operator
**Recommendation:** Option 2 -- restructure to single-schema. The reference project uses this approach successfully. Schema-per-org was chosen for "strong data boundaries" but the mesher project is small enough that FK-based org_id scoping provides equivalent isolation with dramatically simpler query code. This also eliminates the entire `src/org/schema.mpl` provisioning mechanism and `src/db/tenant.mpl` helper.
**Warning signs:** Trying to use `Repo.insert` or `Repo.all` within a manually managed transaction.

### Pitfall 5: Migration Function Visibility
**What goes wrong:** Migration functions not marked `pub` are not found by the migration runner.
**Why it happens:** `meshc migrate up` expects `pub fn up(pool :: PoolHandle) -> Int!String`.
**How to avoid:** Always use `pub fn up` and `pub fn down` in migration files.
**Warning signs:** "migration function not found" errors.

### Pitfall 6: Query.where for Non-String Types
**What goes wrong:** `Query.where(:field, value)` only works when the value type matches what the column/query expects.
**Why it happens:** The API is designed around string equality comparisons.
**How to avoid:** Use `Query.where_raw` for comparisons involving type casts, functions, boolean expressions, or NULL checks. The reference project uses `Query.where` only for simple string equality (`Query.where(:status, "unresolved")`, `Query.where(:email, email)`, `Query.where(:fingerprint, fingerprint)`) and `Query.where_raw` for everything else.
**Warning signs:** Unexpected query results or type mismatch errors.

### Pitfall 7: select_raw Required for UUID/Timestamp Columns
**What goes wrong:** Default SELECT returns raw database types. UUID and TIMESTAMPTZ values may not serialize correctly to String without explicit casting.
**Why it happens:** The Row deriving maps through `Map<String, String>` text protocol but column types may not automatically cast.
**How to avoid:** Use `Query.select_raw(["id::text", "created_at::text", ...])` to explicitly cast UUID and timestamp columns to text. The reference project does this consistently for all queries with UUID/TIMESTAMPTZ columns.
**Warning signs:** Empty or garbled values for UUID/timestamp fields.

### Pitfall 8: Current Project Schema Differences from Reference
**What goes wrong:** Blindly copying reference project patterns without accounting for schema differences.
**Why it happens:** The current mesher project has different table structures than the reference:
  - Current: `id TEXT PRIMARY KEY` (client-generated UUIDs via `Crypto.uuid4()`)
  - Reference: `id UUID PRIMARY KEY DEFAULT gen_random_uuid()` (server-generated UUIDs)
  - Current: `users` has `password_salt TEXT`, `updated_at TIMESTAMPTZ` columns
  - Reference: `users` omits both (salt handled by bcrypt, no updated_at)
  - Current: `organizations` has `owner_id`, `schema_name` columns
  - Reference: `organizations` has `slug` instead, no owner_id/schema_name
  - Current: Schema-per-org with separate `projects`/`api_keys` in tenant schemas
  - Reference: Single public schema, `projects` has `org_id` FK
  - Current: `api_keys` stores `key_hash`/`key_prefix` (hashed storage)
  - Reference: `api_keys` stores `key_value` (plaintext -- design choice for mesher's monitoring keys)
  - Current: Has `invites` and `password_reset_tokens` tables (not in reference)
**How to avoid:** Define models and migrations based on the current project's ACTUAL schema needs, not the reference. Use reference only for ORM API patterns.

```mesh
# AFTER (with ORM + ? operator):
# Note: ? operator only works in functions returning Type!String, not Response.
# Handlers returning Response still need case matching on the top-level result.
# Strategy: Extract a helper function returning Result, call from handler.

fn do_create_org(pool :: PoolHandle, user_id :: String, name :: String) -> String!String do
  let fields = %{"name" => name, "slug" => slugify(name), "owner_id" => user_id}
  let row = Repo.insert(pool, Organization.__table__(), fields)?
  let org_id = Map.get(row, "id")
  let _ = Repo.insert(pool, OrgMembership.__table__(), %{"user_id" => user_id, "org_id" => org_id, "role" => "owner"})?
  Ok(org_id)
end

pub fn handle_create_org(pool, request) -> Response do
  let user_result = extract_user_id(pool, request)
  case user_result do
    Err(_) -> HTTP.response(401, json { error: "unauthorized" })
    Ok(user_id) -> do
      let body = Request.body(request)
      let parsed = Json.parse(body)
      case parsed do
        Err(_) -> HTTP.response(400, json { error: "invalid JSON body" })
        Ok(body_json) -> do
          let name = Json.get(body_json, "name")
          let result = do_create_org(pool, user_id, name)
          case result do
            Err(e) -> HTTP.response(500, json { error: "failed to create organization" })
            Ok(org_id) -> HTTP.response(201, json { id: org_id, name: name })
          end
        end
      end
    end
  end
end
```

## Architectural Decisions

### Decision 1: Single-Schema (Recommended) vs Schema-Per-Org

**Current:** Schema-per-org (org_{uuid} schemas with separate projects/api_keys tables per schema)
**Recommended:** Single public schema with org_id FK on all org-scoped tables

**Rationale:**
1. The reference project successfully uses single-schema with org_id FKs
2. Eliminates all `with_org_schema`, `provision_org_schema`, `schema_name_for_org` code
3. Enables full ORM usage (no manual transaction/search_path management)
4. Simplifies migrations (one set of tables, not per-org schema provisioning)
5. The current project has only 2 tenant-scoped tables (projects, api_keys) -- the overhead of schema-per-org is disproportionate
6. Security/isolation can still be enforced via WHERE org_id = $1 on every tenant-scoped query

**Impact:** This changes the data model. Projects and API keys move from tenant schemas to public schema with an `org_id` FK. The `organizations.schema_name` column is dropped. The `with_org_schema` helper is deleted. All handlers that used `with_org_schema` are simplified.

**If the user rejects this:** Keep schema-per-org but use `Repo.query_raw`/`Repo.execute_raw` inside the `with_org_schema` callback instead of `Pg.query`/`Pg.execute`. This is a partial ORM conversion -- still cleaner than raw Pool/Pg calls but retains the manual transaction pattern.

### Decision 2: ID Generation Strategy

**Current:** `Crypto.uuid4()` client-side, TEXT PKs
**Recommended:** UUID column type with `DEFAULT gen_random_uuid()` server-side

**Rationale:**
1. Reference project uses UUID columns with server-side generation
2. UUID type is more space-efficient than TEXT (16 bytes vs 36+ bytes)
3. Server-side default means `Repo.insert` can omit the id field and get it back in the returned row
4. All current client-side UUID generation (`let id = Crypto.uuid4()`) can be removed

**Trade-off:** Requires `?::uuid` casting in `Query.where_raw` calls. But the reference project shows this is a well-understood pattern.

### Decision 3: Model File Organization

**Recommended:** Dedicated `types/` directory (matching reference project)

**Rationale:**
1. Reference project uses `types/` with logical grouping: `user.mpl`, `project.mpl`, `issue.mpl`
2. Models are imported by both handlers and storage/queries -- co-locating with either creates circular dependencies
3. Types are stable and change infrequently -- separate directory communicates this
4. Multiple related structs per file (User + OrgMembership + Session in `types/user.mpl`)

### Decision 4: Conversion Order

**Recommended order:**
1. **Types/models** (creates foundation -- no dependencies on existing code)
2. **Migrations** (delete old, create new with ORM DSL -- clean slate)
3. **Storage/queries** (centralized query module -- all Repo/Query calls)
4. **src/auth/cookies.mpl** (shared auth helper -- simple, one SQL query)
5. **src/auth/session.mpl** (login/logout -- imports from queries)
6. **src/auth/reset.mpl** (password reset -- imports from queries)
7. **src/auth/oauth.mpl** (OAuth -- imports from queries)
8. **src/org/schema.mpl** (tenant provisioning -- simplified or deleted based on Decision 1)
9. **src/db/tenant.mpl** (with_org_schema -- simplified or deleted based on Decision 1)
10. **src/org/handlers.mpl** (org CRUD -- deepest nesting, most cleanup)
11. **src/org/invites.mpl** (invite CRUD -- significant nesting)
12. **src/project/projects.mpl** (project/API key CRUD -- uses with_org_schema, most impacted by Decision 1)
13. **main.mpl** (entry point -- may need pool initialization changes, verify routes)

**Rationale:** Bottom-up: create types first (no deps), then the query layer (depends on types), then convert handlers one at a time (each depends on query layer). Auth modules first because they are simpler and establish the pattern. Org/project modules last because they depend on tenant isolation decisions.

### Decision 5: Validation Placement

**Recommended:** Keep validation in handlers (matching current pattern and reference project)

**Rationale:**
1. The reference project has no model-level validation -- validation lives in handler/service code
2. Validation logic (name length, email format) is presentation-layer concern
3. Database constraints (NOT NULL, UNIQUE, FK) provide data-layer validation
4. Moving validation into models adds complexity without clear benefit at this stage

## Raw SQL That Must Remain Raw

Based on reference project ORM boundary analysis, these patterns in the current codebase CANNOT be converted to pure ORM and will need `Repo.query_raw` or `Repo.execute_raw`:

| Pattern | Current Location | Why Raw | ORM API Used |
|---------|-----------------|---------|--------------|
| `crypt($1, gen_salt('bf'))` password hashing | session.mpl, reset.mpl, oauth.mpl | Server-side function in SELECT | `Repo.query_raw` for hash, then `Repo.insert` |
| `crypt($1, password_hash)` auth check | session.mpl | Server-side function in WHERE | `Query.where_raw` within Query builder |
| `SET LOCAL search_path` (if keeping schema-per-org) | tenant.mpl, schema.mpl | Connection-level setting | Stays as `Pg.execute` inside transaction |
| `CREATE SCHEMA IF NOT EXISTS` (if keeping schema-per-org) | schema.mpl | DDL operation | `Repo.execute_raw` or `Pg.execute` |
| `CREATE EXTENSION IF NOT EXISTS pgcrypto` | migration | DDL/extension | `Pool.execute` in migration |
| `NOW() + INTERVAL` for session/token expiry | session.mpl, oauth.mpl | Server-side interval arithmetic | Two-step: `Repo.query_raw` for timestamp, then `Repo.insert` |
| `SET column = NOW()` in updates | invites.mpl, reset.mpl | Server-side function in SET | `Repo.execute_raw` (Map cannot express NOW()) |

## Open Questions

1. **Schema-per-org vs single-schema: user sign-off needed**
   - What we know: Single-schema is dramatically simpler and fully ORM-compatible. Reference project uses single-schema successfully.
   - What's unclear: Whether the user specifically wants to KEEP schema-per-org for future multi-tenant SaaS compliance or is willing to adopt single-schema.
   - Recommendation: Present as Decision 1 in the plan. The planner should propose single-schema as the default approach and explain the trade-offs. If user objects, fallback to schema-per-org with raw SQL inside `with_org_schema`.

2. **organizations.owner_id vs org_memberships role='owner'**
   - What we know: Current schema has `organizations.owner_id` column. Reference project has no owner_id -- ownership is tracked via `org_memberships.role = 'owner'`.
   - What's unclear: Whether to keep the dedicated column or use relationship-based ownership.
   - Recommendation: Keep `owner_id` on organizations for now -- it's a simple FK and avoids the query cost of finding the owner via JOIN. Can always be removed later.

3. **password_salt column**
   - What we know: Current schema has `password_salt TEXT NOT NULL` on users. bcrypt's `gen_salt()` embeds the salt in the hash, making a separate column unnecessary.
   - What's unclear: Whether the code actually uses password_salt anywhere meaningful.
   - Recommendation: Drop `password_salt` column in the new migration. bcrypt via `crypt(password, gen_salt('bf'))` handles salting internally.

4. **API key storage: hashed vs plaintext**
   - What we know: Current project stores `key_hash` + `key_prefix`. Reference stores `key_value` (plaintext).
   - What's unclear: Which approach the user prefers.
   - Recommendation: Keep hashed storage (more secure). This means API key lookup needs `Repo.query_raw` with `WHERE key_hash = $1` after client-side SHA-256 hashing.

5. **HTTP.response_with_headers availability**
   - What we know: Current session.mpl uses `HTTP.response_with_headers` for Set-Cookie. The reference project does not use cookies (API key auth only).
   - What's unclear: Whether `HTTP.response_with_headers` is still the correct API.
   - Recommendation: Keep using `HTTP.response_with_headers` as-is -- this is not a data access concern and does not change during ORM conversion.
