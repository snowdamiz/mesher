Org/project-relevant query functions:
```mesh
# Organization
pub fn insert_org(pool :: PoolHandle, name :: String, slug :: String) -> String!String

pub fn list_user_orgs(pool :: PoolHandle, user_id :: String) -> List<Map<String, String>>!String

pub fn add_member(pool :: PoolHandle, org_id :: String, user_id :: String, role :: String) -> String!String
pub fn check_membership(pool :: PoolHandle, org_id :: String, user_id :: String) -> Map<String, String>!String

# Invites
pub fn create_invite(pool :: PoolHandle, org_id :: String, email :: String, token :: String, invited_by :: String) -> String!String
pub fn get_pending_invite(pool :: PoolHandle, token :: String) -> Map<String, String>!String
pub fn check_pending_invite_by_email(pool :: PoolHandle, org_id :: String, email :: String) -> Bool!String
pub fn check_existing_member_by_email(pool :: PoolHandle, org_id :: String, email :: String) -> Bool!String
pub fn accept_invite(pool :: PoolHandle, invite_id :: String) -> Int!String
pub fn revoke_invite(pool :: PoolHandle, invite_id :: String, org_id :: String) -> Int!String
pub fn list_invites(pool :: PoolHandle, org_id :: String) -> List<Map<String, String>>!String

# Projects
pub fn create_project(pool :: PoolHandle, org_id :: String, name :: String) -> String!String
pub fn list_projects(pool :: PoolHandle, org_id :: String) -> List<Map<String, String>>!String

# API Keys
pub fn create_api_key(pool :: PoolHandle, project_id :: String, key_hash :: String, key_prefix :: String, label :: String) -> String!String
pub fn list_api_keys(pool :: PoolHandle, project_id :: String) -> List<Map<String, String>>!String
pub fn revoke_api_key(pool :: PoolHandle, key_id :: String) -> Int!String
```

CRITICAL: Handlers return Response (not Result) -> CANNOT use ? operator in handlers.
Use explicit case matching on query function results.
CAN use ? in private helper functions that return Type!String.

Slug generation: Use String.replace with spaces->hyphens and lowercase, or a simple slugify helper.
The reference project uses a slug library, but for mesher we can implement a simple inline slug:
  let slug = String.replace(String.lower(name), " ", "-")

**handle_create_org rewrite:**
Current: 13 levels of nesting with manual transaction (Pool.checkout -> Pg.begin -> insert org -> insert membership -> Pg.commit -> Pool.checkin -> provision_org_schema). This is the worst nesting in the codebase.

New version:
1. Extract user_id (keep existing extract_user_id pattern)
2. Parse JSON body, validate name length
3. Generate slug from name: `let slug = String.replace(String.lower(name), " ", "-")`
4. Call `insert_org(pool, name, slug)` — returns org_id (UUID generated server-side)
5. Call `add_member(pool, org_id, user_id, "owner")` — adds creator as owner
6. Return 201 with org_id, name, slug (NO schema_name in response — eliminated)

NO provision_org_schema call. NO manual transaction. The ORM insert functions handle errors via Result type. If insert_org succeeds but add_member fails, the org exists without an owner — this is acceptable for Phase 1.1 (a proper transactional wrapper can be added later if needed; the reference project also does sequential inserts without wrapping in a single transaction).

Target nesting: 5 levels max (extract_user_id case -> JSON parse case -> validation if -> insert_org case -> add_member case).

**handle_list_orgs rewrite:**
1. Extract user_id
2. Call `list_user_orgs(pool, user_id)` — returns List of Maps with org fields + role
3. Map rows to JSON response
4. Target nesting: 3 levels

**handle_get_org rewrite:**
1. Extract user_id
2. Extract org_id from URL param
3. Call `check_membership(pool, org_id, user_id)` — returns Map with "role" or Err
4. Call `get_org(pool, org_id)` — returns Map with org fields
5. Combine role from membership + org fields in response
6. Target nesting: 5 levels

**create_invite_handler rewrite:**
Current: 15+ levels of nesting (deepest in codebase). Checks: auth, org membership, owner role, org exists, not already member, no pending invite, then insert + send email.

New version with query functions:
1. Extract user_id
2. Extract org_id from param
3. Parse JSON body for email
4. Call `check_membership(pool, org_id, user_id)` — check role == "owner"
5. Call `get_org(pool, org_id)` — get org name for email
6. Call `check_existing_member_by_email(pool, org_id, email)` — if true, 409 conflict
7. Call `check_pending_invite_by_email(pool, org_id, email)` — if true, 409 conflict
8. Generate token with Crypto.uuid4()
9. Call `create_invite(pool, org_id, email, token, user_id)`
10. Send invite email
11. Return 201

Target nesting: 6-7 levels (still nested due to sequential checks, but each level is simpler — one function call instead of SQL + result parsing).

**accept_invite_handler rewrite:**
1. Extract token from param
2. Extract user_id
3. Call `get_pending_invite(pool, token)` — returns Map with invite details
4. Call `add_member(pool, org_id, user_id, "member")`
5. Call `accept_invite(pool, invite_id)` (mark as accepted)
6. Return 200

**revoke_invite_handler rewrite:**
1. Extract user_id, org_id, invite_id from params
2. Call `check_membership(pool, org_id, user_id)` — check role == "owner"
3. Call `revoke_invite(pool, invite_id, org_id)`
4. Return 200

**list_invites_handler rewrite:**
1. Extract user_id, org_id
2. Call `check_membership(pool, org_id, user_id)` — any role is fine
3. Call `list_invites(pool, org_id)`
4. Map to JSON response
