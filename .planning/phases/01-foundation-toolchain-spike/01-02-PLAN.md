**Spike 1: spikes/ws_actor_supervision.test.mpl** — Prove that a WebSocket handler actor crash is caught and restarted by the supervisor without affecting other connections.

Test approach:
1. Start a Ws.serve with on_connect, on_message, on_close callbacks
2. In on_message: if message == "crash", call `panic("deliberate crash")` to simulate a handler crash
3. Connect two WebSocket clients (use Http.build or a test helper)
4. Client A sends "crash" — this should crash only Client A's handler actor
5. Client B sends "echo" — this should still receive a response, proving server isolation
6. Verify: Client B's echo response is received; the server did not crash

Use `meshc test` runner with `test` and `describe` blocks. The test may need to spawn the WebSocket server in a background actor and connect via HTTP client. If Mesh's test runner does not support async WS testing natively, write a self-contained main() that:
- Starts WS server on a test port (e.g., 9901)
- Uses Http.build to make requests or uses actors to simulate clients
- Prints PASS/FAIL based on behavior

Reference: Mesh skills/http WebSocket section shows Ws.serve(on_connect, on_message, on_close, port). Each connection is an isolated actor per the HTTP mod.rs docs.

**Spike 2: spikes/pg_set_local.test.mpl** — Prove that SET LOCAL search_path works correctly with connection pooling and does not leak between transactions.

Test approach (requires a running PostgreSQL — use docker compose timescaledb service):
1. Open a Pool with min=2, max=4
2. Create two test schemas: `CREATE SCHEMA IF NOT EXISTS org_test_a` and `CREATE SCHEMA IF NOT EXISTS org_test_b`
3. Create a `data` table in each schema with different content
4. In a transaction with SET LOCAL to org_test_a, query data — verify it returns org_test_a's data
5. After transaction ends, in a NEW transaction with SET LOCAL to org_test_b, query data — verify it returns org_test_b's data (not org_test_a's)
6. Verify: The connection pool correctly resets search_path between transactions (pool.rs auto-rollback on dirty checkin)

The test must:
- Connect to DATABASE_URL from env
- Create test schemas and tables
- Run interleaved queries in transactions
- Assert correct data isolation
- Clean up test schemas

Reference: Pool.checkout/checkin auto-ROLLBACK behavior verified in pool.rs source. SET LOCAL is transaction-scoped by PostgreSQL spec.

**Spike 3: spikes/ws_reconnect.test.html** — Prove that Streem-2's fromWebSocket() automatically reconnects after a server-side connection drop.

Test approach (browser-based — research notes these need a browser environment):
Create a self-contained HTML file that:
1. Imports Streem-2 from a local or CDN path (use a `<script type="module">` tag)
2. Starts a connection via `fromWebSocket('ws://localhost:9902', { reconnect: { maxRetries: 10, initialDelay: 1000, maxDelay: 5000 } })`
3. Displays the `status` signal value in real-time (connecting -> connected -> reconnecting -> connected)
4. Includes a companion Mesh WebSocket server script (or instructions to run one) that:
   - Accepts connections on port 9902
   - After receiving "drop" message, closes the connection from server side
5. Test flow: Connect -> verify status is "connected" -> send "drop" -> verify status goes to "reconnecting" -> verify status returns to "connected" within 5 seconds
6. Display PASS/FAIL on screen with timing information

Also create a small companion Mesh server file `spikes/ws_reconnect_server.mpl` that the tester runs alongside the HTML file:
```mesh
fn on_connect(conn) do Ws.send(conn, "connected") end
fn on_message(conn, msg) do
  if msg == "drop" do
    # Close this connection to simulate server drop
    Ws.send(conn, "dropping")
    # The connection close will happen when handler returns without keeping alive
  else
    Ws.send(conn, msg)
  end
end
fn on_close(_conn) do println("client disconnected") end
fn main() do Ws.serve(on_connect, on_message, on_close, 9902) end
```

Note: This spike may need manual verification (open HTML in browser, observe status transitions). The HTML file should be self-documenting with clear instructions.

**Spike 4: spikes/chart_live_update.test.html** — Prove that LitUI chart pushData() sustains smooth rendering during continuous live data updates.

Test approach (browser-based):
Create a self-contained HTML file that:
1. Imports LitUI chart component (`<script type="module">` importing `@lit-ui/charts/line-chart` or equivalent)
2. Creates a `<lui-line-chart>` element
3. Sets initial data via `prop:data` (JavaScript property, NOT HTML attribute — per anti-patterns in research)
4. Starts a `setInterval` at ~16ms (60fps input rate) calling `chart.pushData(Math.random() * 100)`
5. Measures actual frame rate using `requestAnimationFrame` + `performance.now()` over a 5-second window
6. Reports: average FPS, minimum FPS, total frames, dropped frames
7. PASS criteria: average FPS >= 30 (LitUI uses RAF coalescing so input rate > render rate is expected)

The HTML should display:
- The live-updating chart
- Real-time FPS counter
- PASS/FAIL verdict after 5 seconds of measurement

Reference: LitUI BaseChartElement handles RAF coalescing and Float32Array conversion internally. pushData() is the correct API — do NOT set chart.option after streaming starts (per anti-patterns).

IMPORTANT: Both HTML spike files should include clear instructions at the top as HTML comments explaining how to run them (which servers to start, how to open them).
