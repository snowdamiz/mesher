### Handler pipeline pattern
- Create shared `validate_ingest_request()` wrapper in a separate `src/ingest/middleware.mpl` module
- Wrapper handles: API key auth + rate limit check + body extraction
- Returns validated context (project_id, org_id, body) or error Response
- Each handler calls wrapper once, then implements only its specific parsing logic
- Unify `handle_otlp_logs()` and `handle_otlp_traces()` into a single parameterized function

### Pipe operator adoption (Wave 2)
- Apply pipe operators as a **separate pass** after nesting is flattened
- Use standard `|>` (pipe to 1st argument) everywhere it fits
- Use slot pipe `|N>` (pipe to Nth argument) everywhere it fits — `|2>`, `|3>`, `|4>` etc.
- Apply aggressively: replace all sequential let bindings where a value flows through functions
- This is first real-world adoption of `|N>` — dogfooding for the Mesh language
- Key pattern: many functions take `pool` as arg 1, so `|2>` pipes data into position 2

**Pipe operator semantics (from compiler source):**
- `x |> f(a, b)` desugars to `f(x, a, b)` — value becomes 1st argument
- `x |2> f(a, b)` desugars to `f(a, x, b)` — value becomes 2nd argument
- `x |3> f(a, b)` desugars to `f(a, b, x)` — value becomes 3rd argument
- `|1>` is rejected at parse time (use `|>` instead)
- Chainable: `5 |2> add(10) |> to_string_val` works
- Supports multiline (leading and trailing)

### Build/test strategy
- Build AND test after each module refactoring
- Fix failures immediately before continuing to next module — never leave a broken state
- Commit per module after successful build+test

## Summary

This phase is a behavior-preserving server refactor on top of already working Phase 2 ingest/auth code. The highest-risk surface is in three ingestion handlers (`generic.mpl`, `envelope.mpl`, `otlp.mpl`) where deeply nested auth/rate-limit/parse/store flows can be flattened by extracting Result-returning helpers and reducing top-level handlers to one `case` boundary.

There are three coupled goals that must be planned together: (1) flatten nested functions, (2) deduplicate exact ingest helper logic into `src/ingest/helpers.mpl` and centralize ingest preflight in `src/ingest/middleware.mpl`, and (3) run a second pass to adopt `|>` and `|N>` aggressively. Treat these as separate passes in the plan so behavior parity is validated before syntax-level pipe rewrites.

Baseline checks are green now: `npm run build:server` compiles (`Compiled: server/server`) and `npm run test:server` passes (`4 passed in 2.21s`).

**Primary recommendation:** Plan this as strict module-by-module refactoring with parity gates after each module (`npm run build:server && npm run test:server`), and keep pipe-operator adoption as a dedicated final pass only after flattening+dedupe are stable.

## Architecture Patterns

### Recommended Project Structure
```text
server/src/ingest/
├── helpers.mpl      # shared exact-duplicate JSON/brace/bracket helpers
├── middleware.mpl   # validate_ingest_request wrapper (auth + RL + body)
├── generic.mpl      # generic-specific parsing + pipeline usage
├── envelope.mpl     # envelope-specific parsing + pipeline usage
└── otlp.mpl         # otlp-specific parsing + shared pipeline usage
```

### Pattern 1: Result Helper + Thin Response Handler
**What:** Move deep branch logic into `-> Type!String` helpers and use `?`; handler does a single `case` mapping to HTTP response.
**When to use:** Critical ingest handlers with 4+ nested case/if levels.
**Example:**
```mesh
fn do_oauth_callback(pool, request) -> Response!String do
  let params = extract_oauth_params(request)?
  ...
end

pub fn google_oauth_callback(pool, request) -> Response do
  case do_oauth_callback(pool, request) do
    Err(e) -> guard_error(e)
    Ok(r) -> r
  end
end
```
Source: `server/src/auth/oauth.mpl`.

### Pattern 2: Shared Ingest Preflight Wrapper
**What:** Centralize auth + rate-limit + body extraction in `validate_ingest_request()`.
**When to use:** `handle_generic_event`, `handle_sentry_envelope`, `handle_otlp_logs`, `handle_otlp_traces`.
**Plan impact:** Cuts repeated branch depth and keeps endpoint-specific parsing separate.

### Pattern 3: Exact-Dedupe First, Then Replace Call Sites
**What:** Move only byte-for-byte-equivalent logic into `helpers.mpl` first, then update imports/calls.
**When to use:** `json_field` and brace/bracket matcher helpers.
**Plan impact:** Lower semantic risk than partial/generalized helper rewrites.

### Pattern 4: Pipe Adoption as a Dedicated Wave
**What:** Apply `|>` and `|N>` after flattening has stabilized.
**When to use:** Sequential value-flow `let` chains, especially where `pool` occupies arg 1 and data flows into later args.
**Plan impact:** Makes syntax migration independent from behavior refactors.

### Anti-Patterns to Avoid
- **Flatten + pipe in one edit pass:** makes regressions hard to localize.
- **Changing public handler signatures:** breaks route wiring in `server/main.mpl`.
- **Using `?` in `-> Response` functions:** invalid pattern in this codebase; use Result-returning helpers.
- **Moving non-exact duplicates into shared helpers:** risks subtle behavior changes in ingest parsing.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---|---|---|---|
| Ingest preflight logic | Separate ad-hoc auth/RL/body logic in every handler | Shared `validate_ingest_request()` in `middleware.mpl` | Removes repeated nesting and keeps behavior consistent |
| Dedupe discovery | New parsing abstractions | Exact helper extraction from existing functions | Phase goal is refactor parity, not parser redesign |
| Error propagation in handlers | Custom error monads in `Response` functions | Existing Result helpers + single top-level `case` | Matches known Mesh constraints and existing patterns |
| Regression gating | Ad-hoc manual checks only | Required module-level build+test gates | Prevents broken intermediate states |

**Key insight:** this phase should optimize structure and readability without changing ingest/auth behavior; minimal-delta refactoring patterns are safer than generalized rewrites.

## Common Pitfalls

### Pitfall 1: Scope Drift on "ALL deeply nested functions"
**What goes wrong:** A mechanical scan can flag more than the 15-context target set.  
**Why it happens:** Different nesting definitions (indent depth vs semantic branch depth).  
**How to avoid:** Treat the CONTEXT.md list as locked scope baseline; only expand by explicit discretionary decision.  
**Warning signs:** Planner tasks balloon into non-target helper refactors.

### Pitfall 2: Mesh Function Ordering Breakage
**What goes wrong:** Compilation fails after extraction due to forward references.  
**Why it happens:** Mesh codebase requires bottom-up function ordering.  
**How to avoid:** Insert new leaf helpers before callers in each module.  
**Warning signs:** Build errors about unresolved function names right after extraction.

### Pitfall 3: Invalid `?` Usage in Response Handlers
**What goes wrong:** Type/compile errors when `?` is used in `-> Response` functions.  
**Why it happens:** `?` requires `-> Type!Error` style return.  
**How to avoid:** Use helper `-> X!String` and map once in handler with `case`.  
**Warning signs:** Result type mismatch at handler return points.

### Pitfall 4: Behavior Regressions in Ingest Protocol Semantics
**What goes wrong:** Refactor alters status codes/body conventions (especially Sentry envelope).  
**Why it happens:** Control-flow simplification accidentally changes error arms.  
**How to avoid:** Preserve explicit response mappings and existing helper semantics.  
**Warning signs:** Integration script assertions fail for 200/400/403/429 expectations.

### Pitfall 5: Slot Pipe Misplacement
**What goes wrong:** `|N>` pipes value into wrong argument slot, silently changing behavior.  
**Why it happens:** Many functions take `pool` first; data belongs in arg2+ and must use correct slot.  
**How to avoid:** Apply pipes only after flattening, validate each chain by type/behavior and compile immediately.  
**Warning signs:** Compilation succeeds but tests/behavior diverge in data flow paths.
