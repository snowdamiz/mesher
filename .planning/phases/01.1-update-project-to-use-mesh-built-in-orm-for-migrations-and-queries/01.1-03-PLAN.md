Output: 4 auth files rewritten to import and call query functions, with no inline SQL remaining.
</objective>

CRITICAL Mesh language rules (from Phase 1 decisions):
- Handler functions return Response (not a Result type) -> CANNOT use ? operator directly in handlers
- Must use explicit case matching on Result in Response-returning functions
- CAN use ? operator in helper functions that return Type!String
- Bottom-up ordering required (helpers before public functions)
- Let-binding before if conditions (Mesh parser limitation)
- Case arm expressions must be on same line as ->

**handle_login rewrite:**
- Parse JSON body (keep existing pattern)
- Call `authenticate_user(pool, email, password)` — returns Map with "id" and "email" or Err
- Call `cleanup_expired_sessions(pool)` (fire and forget with `let _ =`)
- Call `create_session(pool, user_id)` — returns token string
- Build Set-Cookie header with the token (not session_id — the token IS the session identifier now)
- Response: 200 with email, Set-Cookie header with token

The login response currently returns `email` in the body. Keep that. The Set-Cookie changes from `mesher_session={uuid}` to `mesher_session={64-char-hex-token}`. Same cookie name, longer value.

Error handling: `authenticate_user` returns Err("not found") for bad credentials. Map this to 401. Session creation Err maps to 500. Use explicit case matching (handler returns Response, cannot use ?).

**handle_logout rewrite:**
- Parse cookie to get session token (keep existing parsing)
- Call `delete_session(pool, token)` instead of inline Pool.execute
- Same response structure (200 + clear cookie)

**auth_middleware rewrite:**
- Parse cookie to get session token
- Call `validate_session(pool, token)` from storage/queries — returns Map with "user_id"
- If Ok, call next(request)
- If Err, respond_unauthorized

IMPORTANT: Flatten nesting where possible. The current handle_login has 5 levels of case nesting. The new version should have at most 3 levels (JSON parse -> authenticate -> session create). Use helper functions to extract common patterns.

Example of flattened handle_login:
```mesh
pub fn handle_login(pool, request) -> Response do
  let raw_body = Request.body(request)
  let parse_result = Json.parse(raw_body)
  case parse_result do
    Err(_) -> HTTP.response(400, json { error: "invalid JSON" })
    Ok(body) -> do
      let email = Json.get(body, "email")
      let password = Json.get(body, "password")
      let auth_result = authenticate_user(pool, email, password)
      case auth_result do
        Err(_) -> HTTP.response(401, json { error: "invalid email or password" })
        Ok(user_row) -> do
          let user_id = Map.get(user_row, "id")
          let user_email = Map.get(user_row, "email")
          let _ = cleanup_expired_sessions(pool)
          let session_result = create_session(pool, user_id)
          case session_result do
            Err(_) -> HTTP.response(500, json { error: "session creation failed" })
            Ok(token) -> HTTP.response_with_headers(200, json { email: user_email }, %{"Set-Cookie" => "mesher_session=" <> token <> "; HttpOnly; Path=/; SameSite=Lax; Max-Age=86400"})
          end
        end
      end
    end
  end
end
```

This is 4 levels of nesting — down from 6 in the original. The key improvement is that `authenticate_user` encapsulates the SQL complexity.

**request_reset_handler rewrite:**
- Parse JSON to get email
- Call `get_user_by_email(pool, email)` — if Err (user not found), still return success response (prevent email enumeration)
- If user found: generate token with Crypto.uuid4(), hash with Crypto.sha256()
- Call `invalidate_existing_reset_tokens(pool, user_id)` (fire and forget)
- Call `create_reset_token(pool, user_id, token_hash)`
- Send email with raw token
- Always return success response (even on errors — security best practice to prevent enumeration)

**confirm_reset_handler rewrite:**
- Parse JSON to get token and new_password
- Hash token with Crypto.sha256()
- Call `validate_reset_token(pool, token_hash)` — returns Map with "id" and "user_id" or Err
- If Err, return 400 (invalid/expired token)
- Call `update_user_password(pool, user_id, new_password)` — this handles the crypt() two-step internally
- Call `mark_reset_token_used(pool, token_id)` (fire and forget)
- Call `delete_user_sessions(pool, user_id)` (force re-login)
- Return 200

NOTE: The current confirm_reset_handler generates a new_salt and passes it to the update. The new version drops password_salt entirely — the bcrypt hash from crypt(password, gen_salt('bf')) includes the salt. The update_user_password function in queries.mpl handles the two-step crypt() pattern internally.

Nesting reduction: Current reset handler has 7+ nesting levels. The new version should have at most 4 (JSON parse -> validate token -> update password -> done).
