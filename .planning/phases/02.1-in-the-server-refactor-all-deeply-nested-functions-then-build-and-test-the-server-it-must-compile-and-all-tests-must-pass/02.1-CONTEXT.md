# Phase 2.1: Refactor ALL deeply nested functions — Context

**Gathered:** 2026-03-03
**Status:** Ready for planning

<domain>
## Phase Boundary

Refactor all deeply nested functions in the server codebase (3+ levels of nesting), consolidate duplicated code across modules, apply pipe operators (`|>` and `|N>`) everywhere they fit, then build and test. The server must compile and all tests must pass. This is a pure refactoring phase — no new features, no new endpoints, no behavior changes.

</domain>

<decisions>
## Implementation Decisions

### Refactoring depth
- Flatten ALL functions with 3+ levels of nesting (15 functions total: 5 critical at 4+ levels, 10 high at 3 levels)
- For ingest handlers (critical nesting): extract helper functions with `-> Type!Error` return annotation and use `?` operator; handler calls helper and matches Ok/Err once
- For auth handlers (simpler 3-level nesting): lighter touch — simplify where obvious, don't force helper extraction if the 3-level nesting is already clear enough
- Recursive functions (process_items, process_log_records, process_trace_exceptions): Claude's discretion on whether internal nesting warrants flattening

### Code deduplication
- Create shared `src/ingest/helpers.mpl` for exact duplicate helpers across ingest files:
  - `json_field()` (duplicated in generic.mpl, otlp.mpl, envelope.mpl)
  - `find_matching_brace()` (duplicated in generic.mpl, envelope.mpl)
  - `find_matching_bracket()` (duplicated in generic.mpl, envelope.mpl)
- Only move exact duplicates — leave file-specific variants in place
- Also consolidate shared patterns in auth modules (extend cookies.mpl if needed)

### Handler pipeline pattern
- Create shared `validate_ingest_request()` wrapper in a separate `src/ingest/middleware.mpl` module
- Wrapper handles: API key auth + rate limit check + body extraction
- Returns validated context (project_id, org_id, body) or error Response
- Each handler calls wrapper once, then implements only its specific parsing logic
- Unify `handle_otlp_logs()` and `handle_otlp_traces()` into a single parameterized function

### Pipe operator adoption (Wave 2)
- Apply pipe operators as a **separate pass** after nesting is flattened
- Use standard `|>` (pipe to 1st argument) everywhere it fits
- Use slot pipe `|N>` (pipe to Nth argument) everywhere it fits — `|2>`, `|3>`, `|4>` etc.
- Apply aggressively: replace all sequential let bindings where a value flows through functions
- This is first real-world adoption of `|N>` — dogfooding for the Mesh language
- Key pattern: many functions take `pool` as arg 1, so `|2>` pipes data into position 2

**Pipe operator semantics (from compiler source):**
- `x |> f(a, b)` desugars to `f(x, a, b)` — value becomes 1st argument
- `x |2> f(a, b)` desugars to `f(a, x, b)` — value becomes 2nd argument
- `x |3> f(a, b)` desugars to `f(a, b, x)` — value becomes 3rd argument
- `|1>` is rejected at parse time (use `|>` instead)
- Chainable: `5 |2> add(10) |> to_string_val` works
- Supports multiline (leading and trailing)

### Build/test strategy
- Build AND test after each module refactoring
- Fix failures immediately before continuing to next module — never leave a broken state
- Commit per module after successful build+test

### Claude's Discretion
- Whether to flatten internal nesting in recursive functions
- Specific helper function names and signatures
- Auth module simplification approach (case by case)
- Exact pipe operator placement decisions per function

</decisions>

- Two distinct waves: Wave 1 (flatten nesting + dedup + shared middleware), Wave 2 (pipe operators)
