<!-- Sentry envelope format from RESEARCH.md -->
Sentry envelope structure:
```
Line 1: {"event_id":"...","dsn":"...","sent_at":"...","sdk":{"name":"...","version":"..."}}\n
Line 2: {"type":"event","length":1234}\n
Line 3-N: <event JSON payload>\n
(repeat for additional items)
```

Sentry event payload key fields (exception.values[0]):
- type, value, stacktrace.frames[], mechanism
- exception.values is an ARRAY (can have multiple exceptions in a chain)
- frames are ordered oldest-to-newest (bottom of stack first)

**Envelope parsing (helper functions, bottom of call stack):**

1. `fn parse_envelope_header(header_line :: String) -> Map<String, String>!String`
   - Json.parse the first line
   - Extract: event_id, dsn (optional), sent_at, sdk.name, sdk.version
   - Return as flat map

2. `fn parse_item_header(header_line :: String) -> Map<String, String>!String`
   - Json.parse the item header line
   - Extract: type, length (optional)
   - Return as map

3. `fn extract_exception_data(event_json :: String) -> Map<String, String>!String`
   - Parse the event JSON payload
   - Navigate to exception.values[0] (first exception in chain)
   - Extract: type, value, stacktrace.frames (as JSON string)
   - Also extract top-level fields: platform, level, environment, release, server_name, tags, extra, contexts, timestamp
   - Return as flat map with keys matching IngestEvent fields

4. `fn normalize_sentry_frames(frames_json :: String) -> String`
   - Parse frames array from Sentry format
   - For each frame: extract filename, function, in_app
   - Re-serialize as simplified JSON array (just filename, function, in_app -- strip lineno, colno, abs_path)
   - Return JSON string

**Rate limit response helper (for Sentry-compatible headers):**

```
fn rate_limit_response(retry_after :: String) -> Response do
  # Per locked decision: return X-Sentry-Rate-Limits and Retry-After headers on 429.
  # Mesh supports HTTP.response_with_headers -- confirmed in server/src/auth/session.mpl:59.
  let headers = %{
    "Retry-After" => retry_after,
    "X-Sentry-Rate-Limits" => retry_after <> ":error:scope"
  }
  HTTP.response_with_headers(429, json { error: "rate limit exceeded", retry_after: retry_after }, headers)
end
```

**Main handler (top of call stack):**

```
pub fn handle_sentry_envelope(pool :: PoolHandle, request) -> Response do
  # 1. Authenticate: extract_and_validate_api_key(pool, request)
  #    On failure: return HTTP 403 {"error": "invalid API key"}

  # 2. Rate limit check: get_rate_limit_config(pool, org_id), check_rate_limit(...)
  #    If not allowed: return rate_limit_response(retry_after)
  #    This returns 429 with Retry-After and X-Sentry-Rate-Limits headers
  #    using HTTP.response_with_headers (same pattern as Set-Cookie in session.mpl)

  # 3. Parse envelope body:
  #    let body = Request.body(request) (or however Mesh provides request body)
  #    let lines = String.split(body, "\n")
  #    Parse envelope header (line 0)
  #    Iterate through remaining lines in pairs (item header + payload)

  # 4. For each item:
  #    - Parse item header to get "type"
  #    - If type == "event" or type == "error": process the event
  #    - ALL other types (attachment, session, transaction, check_in, statsd,
  #      profile, replay_event, replay_recording, feedback): SILENTLY DISCARD
  #      (locked decision: accept envelope, process errors only, drop rest, return 200)

  # 5. For each error/event item:
  #    a. extract_exception_data from the JSON payload
  #    b. scrub_event_fields(pool, org_id, message, exception_value, stacktrace_json, ...)
  #    c. compute_fingerprint(exception_type, scrubbed_stacktrace_json)
  #       IMPORTANT: fingerprint AFTER scrubbing (per RESEARCH.md)
  #    d. upsert_issue(pool, project_id, fingerprint, title, level)
  #       title = exception_type <> ": " <> exception_value (truncated to 255 chars)
  #    e. insert_event(pool, project_id, issue_id, event_id, ...)

  # 6. Return Sentry-compatible response:
  #    HTTP.response(200, json { id: event_id })
  #    (locked decision: return event_id in response body)

  # Error handling: use explicit case matching throughout (not ? operator)
  # since function returns Response
end
```

**Edge cases to handle:**
- Empty envelope body: return 400
- No processable items (all discarded types): still return 200 (silently discard, locked decision)
- Missing exception data (event without exception): store with empty exception fields, fingerprint on message only
- Multiple event items in one envelope: process all, return the first event_id
- Envelope header has event_id: use it. If missing: generate with Crypto.uuid4()

**Critical Sentry format notes:**
- Sentry envelope uses `\n` (not `\r\n`) as separator
- The `length` field in item headers tells exact byte count of payload
- If `length` is present, read exactly that many bytes (handle embedded newlines)
- If `length` is absent, read until next `\n` (rare for events)
- For Phase 2, simplify: since @sentry/node always includes length for event items, use the length field. But as a fallback, if parsing with length fails, try newline-splitting.
