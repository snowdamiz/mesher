Endpoints:
- POST /api/{project_id}/envelope/ -- Sentry envelope (X-Sentry-Auth or sentry_key param)
- POST /v1/logs -- OTLP logs (Authorization: Bearer <key>)
- POST /v1/traces -- OTLP traces (Authorization: Bearer <key>)
- POST /v1/metrics -- OTLP metrics stub (Authorization: Bearer <key>)
- POST /api/{project_id}/events -- Generic JSON (Authorization: Bearer <key>)
- GET /health/ingest -- Health check (no auth)

Response formats:
- Sentry: {"id": "<event_id>"}
- OTLP: {"partialSuccess":{}}
- Generic: {"id": "<event_id>"}
- Health: {"status":"healthy","db":"ok","rate_limiter":"ok"}
- Auth failure: 403 {"error":"invalid API key"}
- Rate limit: 429 {"error":"rate limit exceeded","retry_after":N}

Create `server/tests/test_ingestion.sh` -- a comprehensive curl-based integration test suite.

**Prerequisites section at top:**
The test script needs a running server with seeded test data. Add a setup section that:
1. Checks if server is running: `curl -sf http://localhost:8080/health || { echo "Server not running"; exit 1; }`
2. Seeds test data via SQL (psql to the local TimescaleDB):
   - Create a test user, org, project via direct SQL INSERT (or use the API)
   - Create an API key with known raw key value (e.g., "test-api-key-12345678") and its SHA-256 hash
   - Store the project_id and API key for use in tests
3. Define helper functions:
   ```bash
   TEST_KEY="test-api-key-12345678"
   TEST_KEY_HASH=$(echo -n "$TEST_KEY" | sha256sum | awk '{print $1}')
   BASE_URL="http://localhost:8080"
   PASS=0; FAIL=0
   assert_status() { ... }  # Check HTTP status code
   assert_body_contains() { ... }  # Check response body
   ```

**Test cases (one function per test):**

1. **test_health_ingest** -- GET /health/ingest returns 200 with status "healthy"

2. **test_sentry_envelope_valid** -- POST valid Sentry envelope to /api/{project_id}/envelope/
   - Send with X-Sentry-Auth header: `Sentry sentry_key=$TEST_KEY, sentry_version=7`
   - Envelope body: envelope header + event item with TypeError exception
   - Assert: 200, response contains "id"
   - Assert: SELECT from events WHERE project_id = ... returns 1 row
   - Assert: SELECT from issues WHERE project_id = ... returns 1 row

3. **test_sentry_dedup** -- Send same error twice, verify 1 issue with event_count=2
   - Send identical envelopes twice
   - Assert: events count = 2, issues count = 1, issue.event_count = 2

4. **test_sentry_auth_sentry_key_param** -- Auth via ?sentry_key=... query parameter
   - POST to /api/{project_id}/envelope/?sentry_key=$TEST_KEY
   - Assert: 200

5. **test_sentry_auth_invalid** -- Invalid API key returns 403
   - Send with X-Sentry-Auth: `Sentry sentry_key=invalid-key`
   - Assert: 403

6. **test_sentry_discard_non_event** -- Envelope with session item is silently accepted
   - Send envelope with `{"type":"session"}` item
   - Assert: 200 (not rejected)

7. **test_otlp_logs** -- POST OTLP JSON LogRecord with exception attributes
   - Send to /v1/logs with Authorization: Bearer $TEST_KEY
   - Body: resourceLogs with one ERROR LogRecord containing exception.type, exception.message, exception.stacktrace
   - Assert: 200, response contains "partialSuccess"
   - Assert: event stored in DB with correct exception_type

8. **test_otlp_traces** -- POST OTLP JSON with exception span event
   - Send to /v1/traces with Authorization: Bearer $TEST_KEY
   - Assert: 200, exception event stored

9. **test_otlp_metrics_stub** -- POST to /v1/metrics returns 200 (no storage)
   - Assert: 200, no rows in events table for this specific payload

10. **test_otlp_protobuf_rejected** -- POST with Content-Type: application/x-protobuf returns 415

11. **test_generic_json_valid** -- POST to /api/{project_id}/events with simplified JSON
    - Assert: 200, event stored

12. **test_generic_json_missing_message** -- Missing required "message" field returns 400

13. **test_environment_tagging** -- Send events with different environment values
    - Send one with "production", one with "staging"
    - Assert: both stored, environments match in DB

14. **test_fingerprint_line_numbers** -- Same error with different line numbers produces same issue
    - Send two events identical except for lineno in frames
    - Assert: 1 issue (not 2)

15. **test_rate_limiting** -- Send rapid events exceeding rate limit
    - If rate limit is 1000/min, this test may need a lower limit for testing
    - OR: Set test org rate limit to 5/min via direct SQL UPDATE
    - Send 6 events rapidly
    - Assert: first 5 return 200, 6th returns 429

**Script structure:**
```bash
#!/bin/bash
set -euo pipefail

# Colors for output
RED='\033[0;31m'; GREEN='\033[0;32m'; NC='\033[0m'

# Setup, helpers, seed data...

# Run all tests
test_health_ingest
test_sentry_envelope_valid
test_sentry_dedup
# ... etc

# Summary
echo "Passed: $PASS / Failed: $FAIL"
[ $FAIL -eq 0 ] || exit 1
```
