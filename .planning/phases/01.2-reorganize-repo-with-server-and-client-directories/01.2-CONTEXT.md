# Phase 01.2: Reorganize repo with server and client directories - Context

**Gathered:** 2026-03-03
**Status:** Ready for planning

<domain>
## Phase Boundary

Reorganize the repository into explicit `server/` and `client/` directory roots while preserving existing product behavior. This phase is structural: it clarifies ownership, paths, and workflows, not feature expansion.

</domain>

<decisions>
## Implementation Decisions

### Directory boundaries
- Use top-level `server/` and `client/` naming.
- Move backend code fully under `server/`:
  - `main.mpl` -> `server/main.mpl`
  - `src/` -> `server/src/`
  - `migrations/` -> `server/migrations/`
- Rename existing `frontend/` directory to `client/`.
- Keep repo-wide planning/knowledge directories at root (`.planning/`, `.gitnexus/`).
- Move server-only spike artifacts from root `spikes/` into a server-owned location as part of this reorg.
- Keep root orchestration ownership for top-level infra coordination.

### Command workflow
- Root-first command entry is the primary developer workflow.
- Use scoped command naming at root (for example `dev:server`, `dev:client`, `build:server`, `build:client`).
- Each app owns its own build/test commands; root can aggregate.
- Dependency management remains app-scoped (isolated dependencies/lockfiles per app where applicable).

### Runtime wiring
- Place backend Docker build file(s) under `server/`; root compose points at `./server` build context.
- Preserve runtime behavior in this phase (no product/runtime mode changes).
- Keep client API calls using relative `/api` paths.
- Rename Docker Compose service `app` to `server` to align with directory semantics.

### Naming consistency
- Standard source layout: `server/src` and `client/src`.
- Keep current Mesh module naming/import semantics initially (no namespace redesign during this structural move).
- Standardize terminology in docs/scripts on `server/client`.

### Claude's Discretion
- Exact root command implementation mechanism (script wrappers vs Make/task runner) as long as root-first scoped commands are preserved.
- Exact destination path for migrated server-only spikes.
- Order of file moves and compatibility shims during migration, provided behavior remains unchanged.

</decisions>

<specifics>
## Specific Ideas

- This phase should deliver clean structure and ownership, not new capabilities.
- Structural changes should minimize behavioral risk by preserving API contracts and client runtime flow.

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `main.mpl`: Current backend entrypoint and HTTP route wiring; primary integration anchor for move into `server/`.
- `src/` modules (`auth`, `org`, `project`, `storage`, `types`, `db`): Existing backend domain organization can be preserved under `server/src/`.
- `frontend/package.json` scripts and Vite setup: Existing client workflow baseline for transition to `client/`.
- `frontend/src/lib/api.ts`: Central API client already standardized on relative `/api` calls.

### Established Patterns
- Backend currently lives at repo root with Mesh files (`main.mpl`, `mesh.toml`, `src/`, `migrations/`).
- Client is currently isolated in its own app directory (`frontend/`) with independent toolchain.
- Root-level orchestration exists via `docker-compose.yml`, `.env.example`, and `Dockerfile`.
- Client routing is hash-based and backend API integration is path-relative, which supports no-behavior-change structural moves.

### Integration Points
- `docker-compose.yml` build context and service naming must be updated for `server/` pathing and `server` service naming.
- `Dockerfile` COPY paths must align with moved backend sources.
- Any scripts/docs referencing `frontend/`, root `src/`, root `migrations/`, or `app` service naming need coordinated updates.

</code_context>
