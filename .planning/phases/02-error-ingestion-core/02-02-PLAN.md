**Step 2 -- If Valkey client exists in Mesh:**
Create `server/src/ingest/ratelimit.mpl` with:

```
# Rate limiting module using Valkey sliding window counters.
# Atomic INCR prevents race conditions under concurrent requests.
# Key format: "rl:{org_id}:{minute_bucket}"
# Falls back to "allow" if Valkey is unavailable (fail-open for availability).

pub fn check_rate_limit(valkey_conn, org_id :: String, limit :: Int) -> Map<String, String>!String do
  # 1. Compute current minute bucket key
  # 2. INCR the key atomically (returns new count)
  # 3. If count == 1, set EXPIRE to 60 seconds
  # 4. Return {"allowed" => "true"/"false", "current" => count_str, "limit" => limit_str, "retry_after" => seconds_str}
end
```

The function signature should accept whatever connection type the Mesh Valkey client provides.

Include a `pub fn connect_valkey(url :: String)` function that opens a Valkey connection using the discovered API.

Include a `pub fn compute_retry_after() -> String` function that returns seconds remaining in the current rate limit window.

**Step 3 -- If Valkey client does NOT exist in Mesh:**
Implement a PostgreSQL-based fallback rate limiter:
- Use a `rate_limit_counters` table (org_id, window_start, count) with INSERT ON CONFLICT DO UPDATE
- OR use an in-memory approach with a module-level mutable state if Mesh supports it
- Document the limitation clearly in comments
- The function signature remains the same so callers don't need to know the backend

**Important patterns:**
- Use `pub fn` for all exported functions
- Bottom-up ordering (helper functions before public functions)
- Single-word module file name (ratelimit, not rate_limit)
- Return Map<String, String> to match established patterns
- Fail-open: if Valkey/DB check fails, allow the request (log warning)
