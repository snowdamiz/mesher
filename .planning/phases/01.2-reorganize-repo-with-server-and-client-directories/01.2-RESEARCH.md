# Phase 01.2: Reorganize repo with server and client directories - Research

**Researched:** 2026-03-03
**Domain:** Repository structure refactor (Mesh backend + Streem/Vite client + Docker orchestration)
**Confidence:** HIGH

## User Constraints

### Implementation Decisions

#### Directory boundaries
- Use top-level `server/` and `client/` naming.
- Move backend code fully under `server/`:
  - `main.mpl` -> `server/main.mpl`
  - `src/` -> `server/src/`
  - `migrations/` -> `server/migrations/`
- Rename existing `frontend/` directory to `client/`.
- Keep repo-wide planning/knowledge directories at root (`.planning/`, `.gitnexus/`).
- Move server-only spike artifacts from root `spikes/` into a server-owned location as part of this reorg.
- Keep root orchestration ownership for top-level infra coordination.

#### Command workflow
- Root-first command entry is the primary developer workflow.
- Use scoped command naming at root (for example `dev:server`, `dev:client`, `build:server`, `build:client`).
- Each app owns its own build/test commands; root can aggregate.
- Dependency management remains app-scoped (isolated dependencies/lockfiles per app where applicable).

#### Runtime wiring
- Place backend Docker build file(s) under `server/`; root compose points at `./server` build context.
- Preserve runtime behavior in this phase (no product/runtime mode changes).
- Keep client API calls using relative `/api` paths.
- Rename Docker Compose service `app` to `server` to align with directory semantics.

#### Naming consistency
- Standard source layout: `server/src` and `client/src`.
- Keep current Mesh module naming/import semantics initially (no namespace redesign during this structural move).
- Standardize terminology in docs/scripts on `server/client`.

### Claude's Discretion
- Exact root command implementation mechanism (script wrappers vs Make/task runner) as long as root-first scoped commands are preserved.
- Exact destination path for migrated server-only spikes.
- Order of file moves and compatibility shims during migration, provided behavior remains unchanged.

### Deferred Ideas
- None — discussion stayed within phase scope.

## Summary

This phase is a structural extraction, not a feature phase. Planning should optimize for low-risk path moves, complete wiring updates, and deterministic validation that behavior is unchanged.

The key technical constraint is Mesh project-root behavior: `meshc build` requires `main.mpl` in the target directory root, `meshc deps` expects `mesh.toml` there, `meshc migrate` resolves `migrations/` relative to that directory, and test root resolution walks up to nearest `main.mpl`. That means the backend should become a self-contained Mesh project under `server/`, and root wrappers should invoke Mesh commands against `server/` explicitly.

Client risk is lower: Vite/Streem app internals remain valid after renaming `frontend/` to `client/` as long as path references and root orchestration scripts are updated. API behavior should remain unchanged because the client already uses relative `/api` calls.

**Primary recommendation:** Plan this as a 3-step migration: (1) move files to `server/` + `client/`, (2) update orchestration/config/scripts to new paths and service name `server`, (3) run a strict verification matrix that compares behavior before/after.

## Architecture Patterns

### Pattern 1: Backend as a self-contained Mesh project (`server/`)

**Use:** Move backend runtime assets under `server/` and execute Mesh commands against that directory.

**Why this is mandatory (HIGH confidence):**
- `meshc build <dir>` checks for `<dir>/main.mpl`.
- `meshc deps` checks `<dir>/mesh.toml`.
- `meshc migrate [DIR]` resolves `<dir>/migrations`.

**Planning implication:** include `mesh.toml` and `mesh.lock` in backend move scope, even though only `main.mpl/src/migrations` were explicitly listed.

### Pattern 2: Root as orchestration only

**Use:** keep root for compose, env templates, planning artifacts, and cross-app command entry.

**Planning implication:** root should not become a runtime app root after this phase; it is the coordinator.

### Pattern 3: Client as standalone app (`client/`)

**Use:** rename `frontend/` to `client/` and keep current Vite/Streem app internals.

**Why low risk (HIGH confidence):**
- Current client uses relative `/api` fetches (`frontend/src/lib/api.ts`), matching locked runtime decision.
- Vite proxy config is local to app (`frontend/vite.config.ts`) and path-agnostic once directory references are updated.

### Pattern 4: Spike ownership split

**Use:** move server-only spikes to server-owned location; keep or relocate browser-only spikes separately.

**Recommended classification:**
- Server-owned: `*.mpl`, `*.test.mpl`, `ws_reconnect_server.mpl`.
- Client/web-owned: `*.html` spike pages.

## Don't Hand-Roll

| Problem | Don’t Build | Use Instead |
|---|---|---|
| Backend path resolution | Custom scripts that infer Mesh root heuristically | Explicitly pass backend dir to Mesh commands (`meshc build server`, `meshc migrate server status`, etc.) |
| Migration path handling | Ad-hoc migration file discovery | Mesh built-in `meshc migrate [DIR] ...` conventions |
| Client API base rewrite | Environment-dependent absolute API URLs | Keep relative `/api` calls and existing proxy pattern |
| Dependency orchestration | Hoisted monorepo install tricks | App-scoped dependencies/lockfiles per locked decision |

## Common Pitfalls

1. Running Mesh commands from wrong root after move.
- Symptom: `No 'main.mpl' found` or missing migrations.
- Mitigation: root wrappers must always target `server` directory explicitly.

2. Forgetting `mesh.toml`/`mesh.lock` move when backend moves.
- Symptom: `meshc deps` fails or runs against wrong directory assumptions.
- Mitigation: include these files in backend move scope.

3. Leaving server test spikes at root after `main.mpl` moves to `server/`.
- Symptom: test root discovery/fallback behavior points at wrong project root.
- Mitigation: move server-owned spikes under server-owned path.

4. Renaming compose service `app` to `server` without updating scripts/docs.
- Symptom: broken operational commands or stale docs.
- Mitigation: grep and update service name references in orchestration commands.

5. Accidental behavior change while touching routing/networking.
- Symptom: client auth/API flows fail.
- Mitigation: do not change API routes; keep relative `/api` calls and proxy behavior.

## Recommended Plan Shape

1. **Filesystem migration plan**
- Move backend project into `server/` including Mesh root files.
- Rename `frontend/` to `client/`.
- Relocate server-only spikes.

2. **Orchestration and command plan**
- Update compose build context + service rename (`app` -> `server`).
- Implement root-first scoped commands (wrapper mechanism at planner discretion).
- Update ignore rules and path references.

3. **Verification and cleanup plan**
- Run backend/client/build/compose smoke checks.
- Validate no API behavior drift.
- Remove temporary compatibility shims (if any were introduced).

## Verification Checklist (for plan acceptance)

### Baseline checks before and after migration
- `meshc build server` succeeds (after move; baseline currently `meshc build .` succeeds).
- `npm --prefix client run build` succeeds (after rename; baseline currently `npm --prefix frontend run build` succeeds).
- `docker compose config --services` lists `server`, `timescaledb`, `valkey` (and no `app`).
- Representative server spike test still runs from new location (e.g., a `.test.mpl` that does not require DB state).

### Path/reference integrity checks
- No operational references to old locations in active runtime files (`frontend/`, root `src/`, root `migrations/`).
- No compose operational references to service name `app`.
